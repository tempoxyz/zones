# Zone Interoperability Protocol

This document specifies how wallets and applications route token transfers across the Tempo ecosystem using unified Tempo addresses.

- **TIP ID**: TIP-XXXX
- **Authors/Owners**: Dankrad Feist
- **Status**: Draft
- **Related Specs/TIPs**: TIP-XXXX (Tempo Address Format), Zone Validium Design
- **Protocol Version**: TBD

# Overview

## Abstract

This TIP defines the zone interoperability protocol that enables seamless token transfers between any two addresses in the Tempo ecosystem—whether on mainnet or any zone. Wallets decode the destination Tempo address to determine the target zone, compute the optimal path (direct transfer, deposit, withdrawal, or bridge), and construct the appropriate transaction(s) for the user to sign.

## Motivation

With multiple zones in the Tempo ecosystem, users need a simple way to send tokens to any address without manually managing deposits and withdrawals. A user should be able to paste a Tempo address (e.g., `tz1...` for zone 1) and have their wallet automatically:

1. Determine where the recipient is (mainnet or which zone)
2. Determine where the sender's funds are
3. Compute the path to move tokens from source to destination
4. Present a clear transaction (or sequence) for the user to approve

This abstracts away the complexity of cross-zone transfers while maintaining user control and transparency.

# Specification

## Address Resolution

When a user initiates a transfer to a Tempo address, the wallet first decodes the address:

```python
raw_address, zone_id, is_testnet = decode_tempo_address(destination)

# zone_id = None means mainnet (Tempo L1)
# zone_id = 1, 2, ... means a specific zone
```

The wallet also knows the user's current context:
- **Source zone**: Where the user's wallet is currently connected (mainnet or a zone)
- **Source address**: The user's address on that zone/mainnet

## Transfer Routing

Based on source and destination, the wallet determines the transfer type:

| Source | Destination | Transfer Type |
|--------|-------------|---------------|
| Mainnet | Mainnet | Direct transfer |
| Mainnet | Zone N | Deposit to zone |
| Zone N | Zone N | Direct transfer (same zone) |
| Zone N | Mainnet | Withdrawal from zone |
| Zone N | Zone M | Cross-zone transfer (N→Mainnet→M) |

### Case 1: Direct Transfer (Same Network)

If source and destination are on the same network (both mainnet, or both same zone):

```
User signs: transfer(to, amount)
```

Single transaction, immediate execution.

### Case 2: Deposit (Mainnet → Zone)

If user is on mainnet and destination is a zone:

```
User signs: ZonePortal.deposit(to, amount, memo)
```

Where:
- `to` = decoded raw address from destination
- `amount` = transfer amount
- `memo` = optional user-provided memo (or zero)

The deposit is processed by the zone sequencer in subsequent blocks.

**For encrypted deposits** (privacy-preserving):
```
User signs: ZonePortal.depositEncrypted(amount, keyIndex, encryptedPayload)
```

Where wallet encrypts `(to, memo)` using the sequencer's current encryption key.

### Case 3: Withdrawal (Zone → Mainnet)

If user is on a zone and destination is mainnet:

```
User signs: ZoneOutbox.requestWithdrawal(to, amount, fee, memo, gasLimit, fallback, callbackData)
```

Where:
- `to` = decoded raw address from destination
- `fee` = calculated using `ZoneOutbox.calculateWithdrawalFee(gasLimit)`
- `gasLimit` = 0 for simple transfers, >0 if callback needed
- `fallback` = user's zone address (for bounce-back if callback fails)
- `callbackData` = empty for simple transfers

The withdrawal is batched and processed on mainnet by the sequencer.

### Case 4: Cross-Zone Transfer (Zone N → Zone M)

If user is on zone N and destination is zone M (different zone):

This requires two steps:
1. **Withdraw from Zone N to Mainnet**
2. **Deposit from Mainnet to Zone M**

The wallet presents this as a multi-step flow:

```
Step 1 (on Zone N):
  User signs: ZoneOutbox.requestWithdrawal(userMainnetAddress, amount, fee, ...)

Step 2 (on Mainnet, after withdrawal processed):
  User signs: ZonePortal[M].deposit(destinationAddress, amount, memo)
```

**Important**: The wallet must:
- Clearly show this is a 2-step process
- Estimate total time (withdrawal finality + deposit processing)
- Show total fees (withdrawal fee + deposit fee)

## Wallet Flow

### 1. Parse Destination

```typescript
interface ParsedDestination {
  rawAddress: string;      // 0x... format
  zoneId: number | null;   // null = mainnet
  isTestnet: boolean;
}

function parseDestination(tempoAddress: string): ParsedDestination {
  // Decode per TIP-XXXX (Tempo Address Format)
  const [rawAddress, zoneId, isTestnet] = decodeTempoAddress(tempoAddress);
  return { rawAddress, zoneId, isTestnet };
}
```

### 2. Determine Route

```typescript
interface TransferRoute {
  type: 'direct' | 'deposit' | 'withdrawal' | 'cross-zone';
  steps: TransferStep[];
  estimatedTime: Duration;
  totalFees: TokenAmount;
}

function computeRoute(
  source: { zoneId: number | null; address: string },
  destination: ParsedDestination,
  amount: TokenAmount,
  token: TokenAddress
): TransferRoute {
  const srcZone = source.zoneId;
  const dstZone = destination.zoneId;
  
  if (srcZone === dstZone) {
    // Same network: direct transfer
    return {
      type: 'direct',
      steps: [{ action: 'transfer', to: destination.rawAddress, amount }],
      estimatedTime: '~1 block',
      totalFees: gasEstimate
    };
  }
  
  if (srcZone === null && dstZone !== null) {
    // Mainnet → Zone: deposit
    const portal = getZonePortal(dstZone);
    const depositFee = portal.calculateDepositFee();
    return {
      type: 'deposit',
      steps: [{ 
        action: 'deposit', 
        portal, 
        to: destination.rawAddress, 
        amount,
        fee: depositFee
      }],
      estimatedTime: '~2-10 blocks',
      totalFees: depositFee + gasEstimate
    };
  }
  
  if (srcZone !== null && dstZone === null) {
    // Zone → Mainnet: withdrawal
    const outbox = getZoneOutbox(srcZone);
    const withdrawalFee = outbox.calculateWithdrawalFee(SIMPLE_TRANSFER_GAS);
    return {
      type: 'withdrawal',
      steps: [{
        action: 'withdrawal',
        outbox,
        to: destination.rawAddress,
        amount,
        fee: withdrawalFee
      }],
      estimatedTime: '~batch finality + processing',
      totalFees: withdrawalFee
    };
  }
  
  // Zone N → Zone M: cross-zone (2 steps)
  const outbox = getZoneOutbox(srcZone);
  const portal = getZonePortal(dstZone);
  const withdrawalFee = outbox.calculateWithdrawalFee(SIMPLE_TRANSFER_GAS);
  const depositFee = portal.calculateDepositFee();
  
  return {
    type: 'cross-zone',
    steps: [
      { action: 'withdrawal', outbox, to: source.mainnetAddress, amount, fee: withdrawalFee },
      { action: 'deposit', portal, to: destination.rawAddress, amount: amount - withdrawalFee, fee: depositFee }
    ],
    estimatedTime: '~batch finality + deposit processing',
    totalFees: withdrawalFee + depositFee
  };
}
```

### 3. Present to User

The wallet MUST clearly present:

1. **Source**: Where funds are coming from
2. **Destination**: Decoded address with zone name/ID
3. **Route**: Visual representation of the path
4. **Steps**: Each transaction that needs signing
5. **Fees**: Breakdown of all fees
6. **Time estimate**: Expected completion time

Example UI for cross-zone transfer:

```
┌─────────────────────────────────────────────────────┐
│ Send 100 USDC                                       │
├─────────────────────────────────────────────────────┤
│ From: Zone 1 (Privacy Zone)                         │
│       tz1YourAddress...                             │
│                                                     │
│ To:   Zone 2 (Gaming Zone)                          │
│       tz2RecipientAddress...                        │
├─────────────────────────────────────────────────────┤
│ Route: Zone 1 → Tempo Mainnet → Zone 2              │
│                                                     │
│ Step 1: Withdraw to Mainnet         ~5 min         │
│         Fee: 0.02 USDC                              │
│                                                     │
│ Step 2: Deposit to Zone 2           ~30 sec        │
│         Fee: 0.01 USDC                              │
├─────────────────────────────────────────────────────┤
│ Total fees: 0.03 USDC                               │
│ Recipient receives: 99.97 USDC                      │
│ Estimated time: ~6 minutes                          │
├─────────────────────────────────────────────────────┤
│            [Cancel]    [Start Transfer]             │
└─────────────────────────────────────────────────────┘
```

### 4. Execute Steps

For multi-step transfers, the wallet:

1. Signs and submits step 1
2. Monitors for completion (withdrawal processed on mainnet)
3. Prompts user to sign step 2
4. Monitors for final completion

The wallet SHOULD persist transfer state to handle:
- App closure/restart
- Network interruptions
- User returning later to complete step 2

## Token Compatibility

Cross-zone transfers require the token to be:
1. The zone's gas token (for that zone's portal), OR
2. A bridged token with liquidity on both sides

The wallet MUST verify token compatibility before presenting the route. If a token cannot be transferred to the destination zone, show an error explaining why.

## Error Handling

| Error | User Message |
|-------|-------------|
| Invalid address format | "This doesn't look like a valid Tempo address" |
| Unknown zone ID | "Zone {id} doesn't exist" |
| Token not supported | "{token} cannot be sent to {zone}" |
| Insufficient balance | "You need {amount} but only have {balance}" |
| Withdrawal pending | "You have a pending withdrawal. Complete it first or wait." |

## Security Considerations

1. **Address verification**: Always show the decoded zone and raw address for user verification
2. **Phishing prevention**: Warn if destination zone is unfamiliar or newly created
3. **Fee transparency**: Never hide fees; show exact amounts deducted
4. **Step confirmation**: For multi-step transfers, confirm each step separately
5. **Timeout handling**: Set reasonable timeouts and allow cancellation where possible

# Invariants

1. **Deterministic routing**: Given source and destination, the route MUST be deterministic
2. **Fee accuracy**: Displayed fees MUST match actual transaction costs (within gas estimation variance)
3. **No hidden steps**: All required transactions MUST be shown to the user before signing
4. **Atomic display**: Amount sent + fees MUST equal amount deducted from user
5. **Zone validation**: Wallet MUST verify zone exists before presenting deposit option
6. **Token verification**: Wallet MUST verify token is transferable to destination before proceeding
